# みんなで WA 検定 — 制作ノート（他アプリ流用のための知見集）

このドキュメントは「**みんなで作ってみんなで解く4択クイズ**（waquiz）」を作る中で
決めたこと・工夫したことを、**別のアプリでも再利用できる形**でまとめたものです。
特に次の2点を厚めに残しています。

- 🅰 **4択クイズ アプリの作り方**（他アプリでも横展開したい中核）
- 🅱 **毎日ログインしてもらうための仕掛け**（リテンション設計）

---

## 0. 全体像（アーキテクチャの型）

| 要素 | 採用したもの | ねらい・流用ポイント |
|---|---|---|
| フロント | 素の HTML / CSS / バニラJS（ビルドなし） | 依存が少なく、GitHub Pages 等にそのまま置ける。学習コスト低 |
| ホスティング | GitHub Pages（静的） | 無料・URL共有だけで配布。**設定リポジトリは非公開のまま、アプリだけ公開リポジトリに置く**のがコツ |
| データ保存 | Supabase（Postgres＋REST、匿名キー） | サーバー運用ゼロで「みんなで共有」を実現。ブラウザから直接読み書き |
| ログイン | Misskey **MiAuth**（ブラウザ完結） | 専用サーバーのアカウント保有者だけ入れる。トークンは端末に保存 |
| デザイン | Bauhaus 風（クリーム地＋赤/黄/青、角丸なし、Archivo/Karla＋Noto Sans JP） | 「1つのデザインシステム＝config＋CSS変数」で全画面を統一 |

**この型の強み**：バックエンドのコードを書かずに「認証・共有・成績・ランキング」まで作れる。
小さなコミュニティ向けアプリを量産するときの雛形にできる。

### ファイル構成
```
waquiz/
  index.html            画面の器（ヘッダー/メイン/フッター/メニューの空箱）
  css/style.css         デザインシステム（CSS変数＝色/フォント。角丸なしが肝）
  js/config.js          ★アプリ固有の設定を1か所に集約（下記）
  js/misskey.js         MiAuth ログイン＋シェア
  js/store.js           データ層（Supabase 呼び出しを全部ここに隔離）
  js/app.js             画面ロジック（ビュー切替・描画）
  supabase/             スキーマ／マイグレーション／シード／リセットSQL
```

### config.js に「アプリ固有」を全集約する
色・ラベル・難易度・オーナー・ハッシュタグ・ログインサーバー等を **config.js だけ**で変えられるようにした。
別アプリへは「config を差し替える → データ層の SQL を流す」で骨格を再利用できる。
```js
window.QUIZ_CONFIG = {
  supabaseUrl, supabaseAnonKey,      // 保管庫の住所と公開キー（anon/publishable は公開OK）
  appName, tagline, shareHashtag,    // 文言
  questionsPerQuiz: 5,               // 1回の出題数
  defaultMisskeyHost, lockMisskeyHost,// 専用サーバー固定
  ranks: [{key,label,color,desc}...],// 難易度（keyは不変、labelは自由）
  daily: {label,color,desc,minStock},// 「本日の問題」
  owners: ['@user@host'],            // 全問題を見られる特別権限
};
```

## 🅰 4択クイズ アプリの作り方（再利用したい中核）

### 1. データモデル（これが土台）
`questions` テーブル1つが本体。**選択肢は配列＋正解インデックス**で持つ。
```
questions(
  id, rank, body,
  choices jsonb,          -- ["正解","不正解1","不正解2","不正解3"]
  correct_index int,      -- 保存時は 0（先頭＝正解）で統一
  explanation text,       -- 解答解説
  scheduled_date date,    -- 「本日の問題」用（通常は NULL）
  created_by / created_by_name / updated_by / updated_by_name,
  created_at / updated_at
)
```
付随テーブル：`comments`（コメント/補足）, `history`（作成/修正の記録）,
`answers`（1問ごとの正誤）, `results`（1セットの成績）, `logins`（ログイン日）。

### 2. 作問 UX の工夫
- **「正解1つ・不正解3つ」を分けて入力**させる（どれが正解かの選択ミスが起きない）。
  保存時に `choices=[正解, 誤, 誤, 誤]` / `correct_index=0` に正規化。
- **出題時に選択肢をシャッフル**（毎回並びが変わる＝答えの位置で覚えられない）。
  表示用に順序を作り、`正解テキストがシャッフル後どこに来たか`を正解位置として判定する。
  → データは「先頭が正解」で固定、**ランダム性は表示層だけ**に閉じ込めるのがポイント。
- **カテゴリ（難易度）と特別枠（本日の問題）を1テーブルで両立**：
  `scheduled_date` が入っていれば「本日の問題」、NULL なら通常難易度プール。
  通常プールの取得は `scheduled_date is null` で絞る（特別枠が混ざらない）。

### 3. 出題〜結果の流れ
- **1プレイ＝5問セット**（`sampleQuestions(rank, 5)` でプールからランダム抽出）。
- 各問：2×2グリッドで回答 → 即・正誤色分け（正解＝そのランクのテーマ色、誤答＝黒枠、他＝グレー）→ 解説表示 → 次へ。
- **結果画面はアニメーションする円グラフ**（SVG円の `stroke-dashoffset` を遷移＋数字カウントアップ）。達成感が出る。
- **正解は含めずに Misskey へシェア**（問題文＋選択肢＋「挑戦してみて↓」＋URL＋ハッシュタグ）。拡散導線。

### 4. みんなで作る／権限モデル（重要な学び）
- 認証は Misskey 側。Supabase は匿名キーで読み書き（内輪向けの緩いRLS）。
- **「本人だけ編集・削除」はアプリ側で制御**（作成者ハンドルと一致する時だけ編集/削除ボタンを出す）。
  ※ 匿名キー方式なので**サーバーで強制はできない**。本当に厳密にしたい場合は
  Supabase Auth や Edge Function を挟む必要がある、という判断メモ。
- **オーナーアカウント**（config.owners）だけ「全員の問題を閲覧」できる特別ビューを用意。
  通常ユーザーは自分の作った問題のみ閲覧。
- コメント／補足は誰でも可（＝共同編集感）。編集だけ本人限定に切り分けた。

### 5. つまずき＆対処（他アプリでも再発しやすい）
- **GitHub Pages が private リポジトリでは無料公開できない** → アプリ専用の**公開リポジトリを別に作る**。
  設定・個人情報を含む本体リポジトリは非公開のまま保てる。
- **アップロード時にフォルダごと入れて `index.html` が1階層深くなり 404** → ルート直下に置く。
- **RLSにDELETE許可を入れ忘れると削除が効かない**（読取・追加はできるのに消せない）。
  → schema に delete ポリシーを明示。**「できる操作＝ポリシーを書いた操作だけ」**と覚える。
- SQLの `42601 / LINE 0` は**空実行**（コメント行だけ貼った等）。
  → **複数マイグレーションは1本にまとめて冪等化**（`add column if not exists` / `create table if not exists` /
  `drop policy if exists → create policy`）すると事故が減る（`migrate_all.sql`）。
- 非エンジニア向けの手順は「倉庫＝Supabase」「合いカギ＝公開キー」等の**比喩＋コピペだけ**で書く。

## 🅱 毎日ログインしてもらうための仕掛け（リテンション設計）

「また明日も開きたくなる」ための要素を複数、**軽い実装**で積んだ。他アプリに移植しやすい順で記載。

### 1. ログインボーナス＝連続ログインストリーク（最重要・最軽量）
- 仕組み：`logins(user_handle, login_date)` に**1日1行**だけ記録（`unique(user_handle, login_date)`＋upsert）。
- 起動時に当日を記録し、**日付集合から「今日から遡って連続何日か」**を計算して
  「🔥 N日連続ログイン中！」をトースト表示（同じ日は端末ごとに1回だけ演出）。
- マイページに「連続◯日／最長◯日／累計◯日」を掲示。
- 学び：**サーバーに持たせるのは"日付だけ"**。連続数・最長は取得後にクライアントで計算すればよい
  （月またぎ・当日未記録・途切れの判定はローカルの日付演算で完結）。
- 発展案（未実装）：連続日数でポイント/称号、カレンダーにスタンプ、本日の問題と連動でボーナス。

### 2. 「本日の問題」＝毎日変わる来訪理由
- `scheduled_date` にその日を設定した問題を、**その日だけ**出題（基本5問ランダム）。
- 作問時に**出題予定日を指定**（既定＝翌日）。運営が前日までに仕込める。
- **在庫が少ない（3問以下）と自動でグレーアウト＋「問題が揃っていません」**（中途半端な出題を防ぐ）。
- 効果：「今日のぶんを解きに来る」習慣づけ。①のストリークと相性が良い。

### 3. 「NEW」バッジ＝更新に気づかせる
- **前回訪問時刻を端末に保存**（localStorage）。次回、それ以降に作られた問題があるカテゴリのボタンに「NEW」表示。
- **初回ログインでは出さない**（うるさくしない）。開いたら基準時刻を更新し、次の新着まで消える。
- 学び：更新通知は**サーバー通知が無くても「前回時刻 vs 最新作成日時」の比較だけ**で作れる。

### 4. 参加のハードルと動機づけの両立
- **入門編だけゲスト（未ログイン）でも解ける**。ただしゲスト導線は控えめにし、**初回は必ずログイン画面**を出す。
  → 「まず1問触れる」体験は残しつつ、基本はログインへ誘導。
- **成績・ランキング・振り返り**（マイページ）で「続ける意味」を可視化。
  ランキングは正答数の合計順＝**たくさん解くほど上位**（毎日コツコツが効く設計）。
- **作問・解答・シェア**が全部 Misskey に流れる → コミュニティのタイムラインが賑わい、また戻ってくる。

### 5. リテンション施策の移植チェックリスト
- [ ] ログイン日を1日1行で記録する `logins` 相当のテーブル
- [ ] 起動時：当日記録 → 連続日数を計算 → 1日1回だけ演出
- [ ] 「今日限定コンテンツ」（本日の問題）＋予約投入＋在庫ガード
- [ ] 「前回時刻 vs 最新更新」で NEW バッジ
- [ ] マイページで成果を可視化（成績・順位・履歴）
- [ ] 成果を外部SNSへシェアできる導線（正解などのネタバレは除く）

## 参考：Misskey MiAuth（ブラウザ完結ログインの要点）
1. セッションUUIDを作り `https://{host}/miauth/{uuid}?name=...&callback=...&permission=write:notes` へ遷移。
2. 戻り先で `POST /api/miauth/{uuid}/check` → `token` と `user` を取得し localStorage 保存。
3. シェアは `https://{host}/share?text=...` を開くだけ（トークン不要で確実）。
4. **サーバーを固定**すれば「そのサーバーのアカウント保有者だけ」ログイン可能にできる。

## 参考：Supabase を使うときの勘所
- anon/publishable キーは**公開してよいキー**。保護は RLS で行う（内輪なら緩め）。
- できる操作＝**ポリシーを書いた操作だけ**（select/insert/update/delete を明示）。
- スキーマ変更は**冪等な1本のマイグレーション**にまとめると非エンジニア運用でも事故りにくい。
